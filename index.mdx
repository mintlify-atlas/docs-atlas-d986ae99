title: "RedSun: Windows Defender LPE — exploit documentation"
description: "RedSun is a proof-of-concept exploit that abuses Windows Defender's cloud-tag file restoration behavior to overwrite System32 files and escalate to SYSTEM."
---

RedSun is a security research proof-of-concept that exposes a logic flaw in Windows Defender: when the antivirus detects a file bearing a cloud sync tag, it rewrites that file back to disk rather than removing it. This PoC weaponizes that behavior through a chain of Windows internals primitives — oplocks, Volume Shadow Copies, the Cloud Files API, and directory reparse points — to achieve an arbitrary write to `C:\Windows\System32` from a low-privilege user account, ultimately replacing `TieringEngineService.exe` and escalating to `NT AUTHORITY\SYSTEM`.

<Warning>
  RedSun is published for educational and defensive security research purposes only. Running this exploit against systems you do not own or have explicit permission to test is illegal and unethical. Understand your local laws before proceeding.
</Warning>

## The exploit at a glance

<Steps>
  <Step title="Write EICAR and tag the file">
    The PoC writes the standard EICAR antivirus test string to a temporary file and opens it with `FILE_EXECUTE` access to trigger Defender's real-time scan. Once Defender responds, the Cloud Files API is used to attach a cloud-sync placeholder tag (`CF_PLACEHOLDER_CREATE_FLAG_MARK_IN_SYNC`) to the file — making Defender treat it as a cloud-backed object to restore rather than delete.
  </Step>
  <Step title="Race Windows Defender with an oplock">
    A batch oplock (`FSCTL_REQUEST_BATCH_OPLOCK`) is acquired on the file inside a Volume Shadow Copy. When Defender triggers its restoration pass, the oplock fires — giving the PoC precise control over the race window.
  </Step>
  <Step title="Redirect the write to System32">
    During the oplock window, the working directory is replaced with a mount-point reparse point targeting `C:\Windows\System32`. Defender's file restoration write lands in System32 instead of the temp directory.
  </Step>
  <Step title="Escalate to SYSTEM">
    `TieringEngineService.exe` in System32 is now replaced with the PoC binary. Triggering the Storage Tiering COM object causes the service to launch the payload as `NT AUTHORITY\SYSTEM`. A named pipe trick spawns `conhost.exe` back in the original user session.
  </Step>
</Steps>

## Key technical primitives

| Primitive | Purpose |
|-----------|---------|
| Cloud Files API (`CldApi.lib`) | Tags the bait file as a cloud placeholder to trigger Defender's restoration logic |
| Batch oplock (`FSCTL_REQUEST_BATCH_OPLOCK`) | Pauses Defender's write at precisely the right moment |
| Volume Shadow Copy detection | Identifies the VSS volume Defender uses to source the restoration copy |
| Directory reparse point | Redirects Defender's write from `%TEMP%` to `System32` |
| COM activation (GUID `50d185b9-...`) | Triggers `TieringEngineService.exe` execution as SYSTEM |
